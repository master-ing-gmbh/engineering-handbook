# Contributing

Thanks for helping improve this handbook.

## Goal

Keep this repo **simple, practical, and current**.

Rules in this repo should be:

- Easy to understand
- Easy to follow
- Based on real team usage at master.ing GmbH
- Minimal — avoid adding rules “just in case”

Follow **KISS** and **YAGNI**:
- Keep it simple
- Don’t add process or standards unless there is a real need

---

## How to Contribute

1. Create a branch
2. Make your change
3. Open a pull request
4. Get review/approval from the agreed maintainers
5. Merge once approved

---

## What Should Be Added

Add guidance only if it is:

- Repeated across projects
- Useful to multiple developers
- Clear and enforceable
- Worth standardizing

**Good examples:**
- Vue component structure
- PHP file naming
- Git branch naming
- PR expectations

**Bad examples:**
- Personal preferences
- One-off project rules
- Hypothetical future standards
- Over-engineered process

---

## Writing Rules

When writing or updating a guide:

- Use Markdown
- Keep it short and direct
- Prefer bullet points over long paragraphs
- Add small examples only when they help
- Avoid vague wording
- Avoid unnecessary theory

Write rules so developers can scan them quickly.

---

## Decision Making

If there is disagreement:

- Prefer the simplest rule
- Prefer what the team already does successfully
- Avoid adding flexibility that creates confusion

If needed, maintainers or tech leads make the final call.

---

## Keep the Handbook Healthy

Before adding something, ask:

- Do we need this now?
- Will more than one project use it?
- Can a new developer understand it quickly?
- Is this a real standard, or just a preference?

If not, leave it out.
